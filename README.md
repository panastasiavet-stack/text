Текущие проблемы (Big Picture)
Запаздывание реакции: команда узнаёт о проблемах от пользователей, а не от мониторинга.

Потеря контекста: логи хранятся локально внутри контейнеров → при перезапуске данные теряются или становятся недоступны.

Фрагментарность данных: нет единого представления о метриках, логах и причинах деградации.

Слепые зоны: периодические перезапуски контейнеров и рост времени ответа не отслеживаются автоматически.

Цель: переход от реактивной отладки к проактивной наблюдаемости.

2. Предлагаемые инструменты и практики

2.1. Централизованный сбор метрик (Prometheus + экспортеры)

Что добавляем:

Prometheus (сбор и хранение метрик)

cAdvisor (метрики контейнеров)

Node Exporter (метрики узлов)

Какие проблемы решает:

Исчезает слепота по CPU, памяти, сети и перезапускам контейнеров.

Появляются временные ряды (latency, error rate, traffic, saturation).

Можно увидеть, что предшествовало перезапуску.

✅ Команда получает объективные данные вместо догадок.

2.2. Централизованный сбор логов (Loki + Promtail)

Что добавляем:

Loki — хранение логов вне контейнеров

Promtail — автоматический сбор логов из контейнеров и узлов

Какие проблемы решает:

Логи не теряются при перезапуске контейнера.

Не нужно заходить в каждый контейнер вручную.

Логи доступны спустя часы и дни после инцидента.

✅ Снижается MTTR (среднее время восстановления).

2.3. Визуализация и единая картина (Grafana)

Что добавляем:

Дашборды с метриками и логами в одном месте

Возможность смотреть логи рядом с графиками

Какие проблемы решает:

Уходит фрагментарность — всё в одном интерфейсе.

Быстро видно аномалии: рост времени ответа + ошибки + перезапуски.

✅ Исчезает необходимость «прыгать» между разными системами.

2.4. Автоматические оповещения (Alertmanager)

Что добавляем:

Алерты на:

высокое время ответа (p95/p99)

рост 5xx ошибок

частые перезапуски контейнеров

недоступность сервиса

Какие проблемы решает:

Команда узнаёт о проблемах раньше пользователей.

Алерты приходят в Telegram/Slack/Email.

✅ Переход от реактива к проактиву.

2.5. Проверки состояния (Probes)

Что добавляем:

Liveness probe — не завис ли контейнер

Readiness probe — готов ли принимать трафик

Startup probe — долгий старт приложения

Какие проблемы решает:

Контейнеры не получают трафик до полной готовности.

Зависшие контейнеры перезапускаются автоматически, но без потери диагностики (логи уже в Loki).

✅ Уменьшаются ложные перезапуски и частичные деградации.

2.6. Сквозная трассировка запросов (Correlation ID)

Что добавляем:

Уникальный идентификатор запроса (Request ID / Correlation ID)

Прокидывание между сервисами

Запись ID во все логи

Какие проблемы решает:

Можно собрать все логи одного запроса, даже если он проходит через несколько контейнеров.

Легко понять, где именно растёт время ответа.

✅ Ускоряется анализ «медленных запросов».
