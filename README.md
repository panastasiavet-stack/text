
1. Недостатки текущей архитектуры
Существующая система страдает от смешивания бизнес-логики с деталями инфраструктуры. Контроллеры и сервисы одновременно выполняют:
⦁ расчёт скидок и проверку льгот (бизнес-правила);
⦁ обработку HTTP-запросов;
⦁ формирование SQL-запросов и работу с ORM;
⦁ конкретное хранение данных.
Это приводит к таким проблемам:
⦁ изменения в базе данных или внешних API требуют правок в бизнес-логике;
⦁ контроллеры становятся перегруженными и выполняют не свои функции;
⦁ тестирование усложняется, так как необходимо поднимать БД, сервер и сторонние сервисы;
⦁ система становится хрупкой, трудной для сопровождения и развития.
2. Рекомендованный архитектурный подход
Оптимальным решением является применение принципов чистой архитектуры (Clean Architecture). Суть подхода:
⦁ бизнес-логика (расчёт скидок, обработка заявок и др.) не должна напрямую зависеть от внешних технологий — HTTP, ORM, SQL, баз данных или API;
⦁ все внешние технологии выносятся на периферию как подключаемые компоненты;
⦁ зависимости направляются внутрь — от инфраструктуры к бизнес-логике, а не наоборот.
3. Как это решает проблему
При внедрении чистой архитектуры система делится на несколько слоёв:
Слой                  | Назначение                 | Пример                                        
----------------------+----------------------------+-----------------------------------------------
Доменный слой         | Бизнес-сущности и правила  | Студент, Заявка, Скидка, Льгота               
Use case (прикладной) | Сценарии использования     | РассчитатьСкидку(), ПроверитьЛьготу()         
Интерфейсные адаптеры | Контроллеры, DTO, мапперы  | HTTP-контроллер, адаптер внешнего API         
Инфраструктура        | ORM, SQL, БД, HTTP-клиенты | Репозиторий на SQLAlchemy, вызов реального API

В результате:
⦁ контроллер лишь принимает запрос и вызывает нужный use case;
⦁ use case работает через интерфейсы (например, IRepository, IExternalApi), а не с конкретными реализациями;
⦁ бизнес-правила можно менять независимо от технологий хранения и передачи данных.
Итог по задаче:
Основная проблема — тесная связь бизнес-логики с инфраструктурой. Решение — изоляция бизнес-ядра с помощью чистой архитектуры и направление зависимостей внутрь. Это позволяет:
⦁ менять БД, ORM и внешний API без изменения бизнес-логики;
⦁ просто тестировать бизнес-логику без реальной инфраструктуры;
⦁ уменьшить связанность и повысить удобство сопровождения системы.
