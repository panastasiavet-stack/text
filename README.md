

**Анализ отказоустойчивости системы бронирования**

**1. Исходные данные (напоминание)**

- Сервис бронирований (BS) масштабируется неограниченно в дата-центре 1 (ДЦ1).
- Сервис ресторанов (RS) представлен единственным экземпляром в дата-центре 2 (ДЦ2), код недоступен, масштабирование не допускается.
- Клиенты — мобильное приложение с жестко зашитым фиксированным списком IP-адресов BS, порядок которых одинаков для всех пользователей.
- Нагрузка:
- В обычном режиме — 100 запросов в секунду.
- В пиковый период (10–15 минут после публикации известной личности) — до 50 000 запросов в секунду.
- Бизнес-требования: отказов быть не должно, всегда есть свободные столики, обработка заявки допускается в течение 24 часов.

Сбой системы происходит из-за двух независимых, но одновременных причин.

---

**2. Две ключевые архитектурные проблемы**

**2.1. Проблема с балансировкой на клиентской стороне**

Механизм сбоя:

1. BS-экземпляры имеют разные IP-адреса, однако в приложении фиксирован один и тот же список.
2. При пике весь трафик идёт на первый IP → первый экземпляр BS выходит из строя.
3. Клиенты переключаются на второй IP → второй экземпляр BS падает.
4. Цикл повторяется, пока не откажут все экземпляры.

Почему это проблема сама по себе:

Несмотря на масштабирование BS (даже до тысячи экземпляров), трафик по-прежнему концентрируется в одном месте, приводя к отказам.

**2.2. «Синхронный шок» единственного сервиса ресторанов (RS)**

Механизм сбоя:

1. Каждый запрос к BS инициирует синхронный REST-запрос к RS.
2. При пике в 50 000 rps даже гипотетически устойчивый BS генерирует такой же поток запросов к RS.
3. Единственный экземпляр RS физически не выдерживает эту нагрузку и рушится.

Почему это отдельная проблема:

Поскольку RS не масштабируется и взаимодействие синхронное, система становится уязвимой к резким нагрузочным всплескам.

---

**3. Предложенная стратегия решения**

**3.1. Решение проблемы клиентской балансировки**

Главное наблюдение:

Обновление приложения с использованием единого DNS-имени не решит проблему — у старых клиентов остается старый фиксированный список IP.

Предлагаемый подход — двойной вход:

| Тип клиента | Точка входа | Обработка в сети |
|----------------------|------------------------------------|------------------------------------------|
| Старая версия (с IP) | IP1, IP2, IP3… (зашиты в приложении) | Все IP перенаправляются на один L7-балансировщик (через NAT, VIP, reverse proxy) |
| Новая версия | api.booking.company (DNS) | DNS указывает на тот же балансировщик |

Результат:

- Старые и новые клиенты направляются на единый балансировщик.
- Балансировщик равномерно распределяет нагрузку между BS-экземплярами (напр., round-robin или leastconn).
- Эффект «домино» устраняется: ни один BS-экземпляр не перегружается.

Реализация: применяется L4/L7 балансировщик (HAProxy, NGINX или облачный ALB) с health checks. Старые IP закрепляются за балансировщиком как дополнительные виртуальные адреса (VIP).

**3.2. Устранение синхронной зависимости от RS**

Решение: перейти на асинхронную модель обработки с использованием очереди и контроля скорости запросов.

Процесс обработки:

1. Клиент отправляет запрос в BS.
2. BS создаёт запись со статусом PENDING и добавляет задачу в очередь.
3. BS сразу возвращает ответ: «Заявка принята, ожидайте подтверждения».
4. Воркер-процессы извлекают задачи из очереди, используя rate limiter.
5. Rate limiter ограничивает поток запросов к RS по допустимому уровню (около 20% от максимума).
6. Воркеры обновляют статус заявки и при необходимости отправляют push-уведомления клиенту.

Почему это эффективно при 50 000 rps:

- Очередь выступает в роли буфера, заявки не теряются.
- Воркеры сглаживают нагрузку, предотвращая «синхронное» перегружение RS.
- Пользователь получает подтверждение заявки, что соответствует бизнес-требованию «отказов не допускать».

Дополнительно:

- Idempotency key защищает от дублирующих запросов.

- Dead Letter Queue
для задач, не обработанных после повторных попыток.
- API для проверки статуса заявки.

**4. Итоговая архитектура (схематично)**

| Проблема | Решение | Обоснование |
|----------------------------------------------|------------------------------------------------------|--------------------------------------|
| Клиентский список IP ведет к точечной нагрузке | Двойной вход с балансировщиком | Равномерное распределение трафика |
| Синхронный REST-запрос к единственному RS | Асинхронная очередь и rate limiting запросов к RS | Предотвращение перегрузки RS |
| Масштабирование BS не решает балансировку | Балансировщик скрывает множественные IP BS | Трафик не концентрируется на одном экземпляре |
| Отказы из-за перегрузок | Буферизация и контроль скорости | Обработка пиковых нагрузок без сбоев |


————————

В приложении зафиксирован список IP-адресов. Для сохранения совместимости старых клиентов необходимо оставить эти IP как точки входа, но перенаправить их на VIP, reverse proxy, NAT или L4/L7 балансировщик нагрузки. Простая замена адреса в новой версии приложения не решает проблему, так как legacy-клиенты продолжат обращаться к старым IP. Поэтому данные IP должны вести не напрямую к отдельным экземплярам сервиса, а в слой балансировки.

Поскольку все клиенты по умолчанию обращаются сначала к первому IP, необходимо ввести Load Balancer или API Gateway, который распределяет нагрузку равномерно между инстансами.

Для исключения ситуаций, когда экземпляры падают один за другим, нужно настроить health checks и выводить из оборота неработающие сервисы. Балансировщик тогда не будет отправлять трафик на отказавшие инстансы.

Актуальная синхронная зависимость сервиса бронирований от сервиса ресторанов следует заменить на асинхронную модель с использованием очереди сообщений. При этом пользовательский запрос не будет ждать ответа от сервиса ресторанов.

Поскольку единственный экземпляр сервиса ресторанов не способен выдержать пиковую нагрузку, очередь с обработкой через воркеры и контролем скорости (rate limiter) позволяет равномерно распределять нагрузку.

Для недопущения отказов пользователям заявки принимаются со статусом PENDING, а клиент получает подтверждение факта приёма заявки вместо отказа.

Чтобы избежать создания дублирующих заявок при повторных запросах, внедряется idempotency key.

Вывод по второму заданию

В системе выявлены две ключевые проблемы, приводящие к сбоям при пиковых нагрузках (50 000 rps):

Проблема 1: Некорректная балансировка из-за зашитого списка IP
⦁ Корень проблемы: мобильное приложение содержит жёстко заданный список IP бронирования с одинаковым порядком у всех клиентов. При пике весь трафик сначала идёт на первый IP → сервис падает → переключение на второй IP → падение второго экземпляра и так далее (эффект домино).
⦁ Почему простой переход на единый DNS не решает проблему: старые версии приложений продолжат обращаться к жёстко зашитым IP.
⦁ Обязательное решение: переназначить старые IP на балансировщик (VIP, Load Balancer, reverse proxy, L4/L7 балансировщик или NAT к балансировщику), чтобы и «наследственные» клиенты попадали на балансировочный слой, который будет равномерно распределять нагрузку.
Проблема 2: Синхронная зависимость от единственного сервиса ресторанов
⦁ Корень проблемы: для каждого запроса сервис бронирования синхронно вызывает REST-сервис ресторанов, который не масштабируется и работает в единственном экземпляре. Пиковая нагрузка в 50 000 rps приводит к его перегрузке и сбоям.
⦁ Решение: переход на асинхронную обработку через очередь и воркеры с rate limiter, ограничивающим скорость вызовов к сервису ресторанов.
⦁ Дополнительные преимущества: пользователи сразу получают подтверждение приёма заявки, нагрузка на сервис ресторанов сглаживается и равномерно
распределяется.

Итог

Внедрение предложенных изменений обеспечивает:

⦁ Обработку высоких пиков нагрузки (до 50 000 rps) без отказов пользователям;
⦁ Защиту единственного сервиса ресторанов от перегрузок благодаря очередям и контролю скорости;
⦁ Корректную работу как с новыми, так и с устаревшими версиями приложения за счёт переназначения старых IP на балансирующий слой;
⦁ Полное соблюдение ограничений — без масштабирования сервиса ресторанов, без изменения его кода и без отказов клиентам.
