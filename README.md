
**Проблемы текущей реализации**

В изначальной версии системы класс CourseManager нарушает принцип единой ответственности, выполняя множество разноплановых задач:

- создание курсов разных типов  
- хранение данных о курсах и пользователях  
- регистрацию студентов  
- расчёт итоговых оценок  
- отправку уведомлений  
- прямое взаимодействие с внешним платежным API  

Вследствие этого любое изменение — добавление нового типа курса, пересмотр формулы оценивания, изменение способа уведомлений или замена платежного провайдера — приводит к необходимости править один и тот же класс.

**Основные архитектурные недостатки**

1. **Перегруженность класса CourseManager** — совмещение создания объектов, бизнес-логики, хранения данных, уведомлений и работы с оплатой в одной сущности.  
2. **Жёсткое создание объектов** — курсы создаются с помощью условных операторов и вызовов конструкторов напрямую.  
3. **Отсутствие отдельного слоя хранения данных** — данные о курсах и пользователях хранятся внутри менеджера.  
4. **Фиксированный алгоритм оценки** — используется единственный метод вычисления итоговых оценок для всех типов курсов.  
5. **Встроенные вызовы отправки уведомлений** — `sendEmail(...)` вызывается прямо внутри методов.  
6. **Прямая зависимость от платежного API** — бизнес-логика связана с конкретной внешней системой оплаты.  
7. **Дублирование логики** — похожие проверки и операции дублируются в разных методах.

**Пример жёсткого создания курсов**

```  
if (type == "online") new OnlineCourse(...)  
else if (type == "offline") new OfflineCourse(...)  
```

**Итог по исходной системе**  
Архитектура плохо масштабируется, слабо тестируется и нарушает принципы DRY, KISS и единой ответственности.

**Новая архитектура**

**Главный принцип**  
Бизнес-логика взаимодействует с интерфейсами, а не с конкретными классами и внешними API.

**Компоненты системы**

| Компонент            | Ответственность                      |
|---------------------|-----------------------------------|
| LearningService     | Координация бизнес-сценариев (создание курсов, запись студентов, расчёт оценок) |
| CourseCreator       | Создание объектов курсов разных типов |
| CourseStorage       | Хранение и получение курсов         |
| UserStorage         | Хранение и получение пользователей  |
| EnrollmentStorage   | Хранение записей о регистрации студентов |
| GradeStorage        | Управление оценками                  |
| PaymentStorage      | Хранение информации о платежах      |
| GradingEngine       | Алгоритмы расчёта итоговых оценок   |
| Notifier            | Отправка уведомлений (email, SMS, мессенджеры) |
| PaymentProcessor    | Внутренний интерфейс оплаты         |
| PaymentGatewayAdapter | Адаптер к внешнему платежному API |

**Структура архитектуры**

1. **Прикладной слой** — `LearningService` реализует ключевые сценарии (создает курсы, регистрирует студентов, рассчитывает оценки, инициирует оплаты), но не хранит данные, не создает курсы напрямую и не вызывает уведомления и внешние API самостоятельно.  
2. **Доменный слой** — содержит сущности предметной области (`Course` с наследниками `OnlineCourse`, `OfflineCourse`, `BlendedCourse`, `User`, `Student`, `Enrollment`, `Grade`, `Transaction`), не зависит от инфраструктуры (БД, HTTP, email).  
3. **Слой хранения данных** — выделенные интерфейсы и реализации для доступа к данным, что позволяет менять БД или ORM без влияния на бизнес-логику.  
4. **Инфраструктурный слой** — реализации конкретных хранилищ, адаптеры платёжного шлюза и отправки уведомлений (EmailNotifier, SmsNotifier, TelegramNotifier), интеграции с внешними сервисами.

- **Factory Method** — для создания курсов без использования условных операторов. Класс `CourseCreator` берет на себя ответственность за порождение объектов, что упрощает добавление новых типов курсов.
- **Strategy** — позволяет переключать алгоритмы оценки с помощью `GradingEngine` (например, `StandardGrading`, `PassFailGrading` и др.).

**Observer — уведомления как реакция на события**  

**Компоненты:** Классы LearningService (поставщик событий) и Notifier (например, EmailNotifier, SmsNotifier, TelegramNotifier).  

**Задача:** Разделить само событие и реакцию на него.  

**Преимущество:** LearningService просто сообщает, что студент зарегистрировался, а подписчики самостоятельно решают, отправлять ли уведомления по email, SMS, Telegram или по всем каналам сразу. Добавление нового способа оповещений сводится к добавлению нового подписчика без изменений в ядре системы.  

---  

**Adapter — слой между бизнес-логикой и внешним платежным сервисом**  

**Класс:** PaymentGatewayAdapter  

**Зачем:** Изолировать LearningService от деталей внешнего API.  

**Плюс:** Если меняется платёжный провайдер (например, с Сбербанка на Stripe или Яндекс.Кассу), достаточно обновить только адаптер. Остальная система продолжит работу без изменений и даже не узнает об этом.  

---  

**Command — каждая операция как отдельный объект**  

**Классы:** CreateCourseCommand, RegisterStudentCommand, PayCourseCommand  

**Цель:** Обернуть действия в независимые объекты.  

**Выгоды:** Операции можно логировать, откладывать выполнение, ставить в очередь, отменять и повторно использовать. Это избавляет от дублирования одинаковых сценариев в разных местах кода.

**Итоги по первому заданию**  

Предлагаемая архитектура устраняет основную проблему исходной системы — сосредоточение разнообразной логики в одном классе `CourseManager`. Вместо этого система разбита на специализированные компоненты с четко разграниченными обязанностями: `LearningService` управляет бизнес-сценариями, `CourseCreator` отвечает за создание курсов, `GradingEngine` реализует расчёт оценок, а `Notifier` занимается отправкой уведомлений.  

Отдельно выделен слой хранения данных. Данные о курсах, пользователях, записях студентов, оценках и платежах не хранятся внутри бизнес-класса, а обрабатываются через интерфейсы хранилищ (`CourseStorage`, `UserStorage`, `EnrollmentStorage`, `GradeStorage`, `PaymentStorage`). Это обеспечивает возможность менять механизм хранения (SQL, ORM, файлы) без влияния на бизнес-логику.
